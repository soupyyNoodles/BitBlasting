Update (A3 - Q2): We have received a few queries about the A3-Q2 submission format. Sharing details below:

You have to submit all the code files in Q2/src, including the ones we have shared with you.
Note: In predict.py, there are some components where you can change code if you need to pass some extra information to your trained model, other than the default ones. The script itself indicates where you can make code changes. Do not change the data loading and evaluation script.

You should not submit trained models and data files. The evaluators will train the model using your code while adhering to the time limits given in the assignment. Finally, the trained models will be used for inference.

You can have different architectures for different datasets/tasks. You can also code them up in separate python scripts. Finally, the compatibility with the downstream prediction and evaluation scripts is what matters.

You may use ensembles as long as the total training time does not exceed the specified limit. Again, you may update the model calling inside the predict.py for that. Finally we will not change the predict command, so make sure you save the model as a single .pt file.

Finally, please remove any temporary and pycache, etc., folders before submitting.

Train command: python train.py --dataset A|B|C --task node|link --data_dir /absolute/path/to/datasets --model_dir /path/to/models --kerberos YOUR_KERBEROS

--model_dir is the absolute path to the directory where your model will be saved. Make sure the model is saved with a name compatible with predict.py so that automated evaluation can take place.

PyTorch GLIBC version issues
Updated 1 day ago by Mridul Gupta
Hi all,
Several of you have been facing old GLIBC version issues. The PyTorch release document (link) clearly mentions that the wheels have been compiled to be manylinux2014 compatible which only requires GLIBC 2.17 or above.


Therefore, I implore you to do the following to resolve this issue:

Ensure that you are not using nightly builds. This can be done by ensuring you use the stable wheel index url from PyTorch installation commands here.
Ensure that you are installing via pip as PyTorch recommends.
Try to use PyTorch 2.5.0+cu118 (or whatever cuda version you need) which is an older version and therefore may be compatible with older GLIBC versions. If the issue still persists, fall back to PyTorch 2.2.0, then fall back to 1.13.
Whichever version you end up using, mention that in a README.md file.
